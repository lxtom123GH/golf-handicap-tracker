# Production Readiness Audit — "What hurts the first real user?"

Read-only audit of `src/` and `functions/index.js`. Findings only — no fixes applied.

Severity key: 🔴 would cause real user harm · 🟡 would cause confusion or data loss · 🟢 polish/improvement

---

## 1. Unhandled errors and silent failures

- `src/competitions.js:42` — `onSnapshot(q, (snapshot) => {...})` for the competitions list has no error callback. If the listener fails (offline, security-rule denial), the competition dropdown silently never populates and nothing tells the user why. 🟡
- `src/competitions.js:203` — `onSnapshot(q, (snapshot) => {...})` for `comp_rounds` likewise has no error callback; the leaderboard silently stays empty/stale on listener failure. 🟡
- `src/practice.js:369` — `onSnapshot(q, (snapshot) => {...})` for `practice_rounds` has no error callback; the practice dashboard silently stays empty if the listener errors. 🟡
- `src/admin.js:133-143` — the "pre-approve email" form's submit handler `await setDoc(...)` is not wrapped in try/catch. A thrown error (e.g. malformed email used directly as a document ID, see Area 2 #5) becomes an unhandled rejection — the form just sits there with no success/error feedback. 🟡
- `src/social.js:50-57` and `src/social.js:74-79` — the "Follow" button handler `await followPlayer(uid)` (and `followPlayer`/`unfollowPlayer` themselves) have no try/catch anywhere in the chain. If `setDoc`/`deleteDoc` fails (offline, permission-denied), it's an unhandled rejection: the button never flips to "✓ Following" and the user sees nothing. 🟡
- `src/ai.js:233` — `getFunctions()` is called with no app/region, so it defaults to `us-central1`, while every Cloud Function is deployed to `australia-southeast1` (see `firebase-config.js:41` and the explicit comment "frontend now targets australia-southeast1 explicitly"). Every "✨ AI Training Plan" / "✨ Lesson Plan" request will fail to resolve the callable function. The surrounding try/catch (line 245) does show a generic error in the modal, so it's not a hard crash — but the AI Coach feature is effectively dead for every user in production despite looking fully wired up. 🔴
- `src/admin.js:163` and `src/admin.js:191` — `loadPreapprovedEmails`/`removePreapprovedEmail` use empty `catch (e) { }` blocks. On failure the list silently stays on "Loading..." or simply doesn't refresh, with zero feedback to the admin. 🟡
- `src/coach.js:73` and `src/coach.js:85` — `loadMyCoaches`/`removeCoach` likewise swallow errors with empty `catch (e) { }`; the "My Coaches" list can get stuck on "Loading..." with no indication anything went wrong. 🟡
- `src/oncourse.js:1452` — inside the penalty-modal click handler, `AppState.currentUser.uid` is accessed directly (no `?.`, unlike the equivalent guards at `oncourse.js:355/483/511`). If the session ends mid-round (token expiry, sign-out in another tab), this throws an uncaught `TypeError`, silently breaking the penalty button with no feedback. 🟡

## 2. Input validation gaps

- `src/whs.js:342` and `src/ui.js:371` — `round.course` is interpolated directly into `tr.innerHTML` with no escaping. The value originates from the free-text "Course Name" field for custom courses (`src/app-v4.js:254-260`, no length limit, no character filtering — only `|| "Custom Course"` as a fallback for emptiness). A user who saves a round with a course name like `<img src=x onerror=...>` produces **stored XSS** that fires for themselves, for any admin browsing via the player select, and for any coach viewing that athlete's rounds. 🔴
- `src/auth-v2.js:59-79` — the registration `displayName` is captured with only an `if (!name)` emptiness check (no length cap, no character/HTML filtering) and stored verbatim in `users/{uid}`. It is later injected unescaped via `innerHTML` in many render paths: `src/admin.js:25`, `src/social.js:47`, `src/social.js:108`, `src/competitions.js:592-593`, `src/notifications.js:70`. Any of these is a **stored XSS** vector that fires in another person's browser (admin dashboard, player search results, following list, competition "regulars" picker) the moment they view a list containing the malicious user. 🔴
- `src/coach.js:36-44` — the "Add Coach" form takes `coach-uid-input`, trims it, and pushes it straight into the user's `coaches` array via `updateDoc` with **no check that the UID corresponds to a real coach** (or any user at all). Garbage strings silently pollute the data model that `array-contains` queries rely on (`src/app-v4.js:376`) to build coach rosters. 🟡
- `src/coach.js:70` — that same unvalidated `cid` string is interpolated both into the visible `<span>` and into an inline `onclick="removeCoach('${cid}')"` attribute. A crafted UID containing a quote or HTML breaks out of the attribute — a self-XSS vector fed by the same unvalidated field above. 🟡
- `src/admin.js:139` and `src/admin.js:238` — the admin "pre-approve" and "invite" flows use the lower-cased, trimmed-but-otherwise-unvalidated email string directly as a Firestore **document ID** (`doc(db, "preapproved_emails", em)`). No email-format check exists; a value containing `/` produces an invalid document path and an uncaught error (Area 1), while other malformed strings create junk documents that the approval lookup (`src/auth-v2.js:65`, exact-match on `email.toLowerCase()`) will simply never match — silent data corruption of the approval list. 🟡
- `src/competitions.js:517-552` — the "Log Competition Round" form only guards `isNaN(score)`; the `comp-date` value flows straight into `new Date(date)` (an empty string yields `Invalid Date`) and is written to Firestore as `payload.date`. No range/sanity checks on `score` (negative values accepted) either. 🟡
- `src/modules/event-binders.js:309` and `:319` — manually-overridable Course Rating / Slope Rating / Daily Handicap inputs injected via `innerHTML` (`oc-manual-cr`, `oc-manual-sr`, `oc-manual-dh`) accept any numeric value (including negative or absurd numbers) with no min/max/sanity validation before they flow into the live-round AGS/Stableford math and are eventually persisted (`src/oncourse.js:1085-1107`). A bad value here can silently corrupt a saved round's `adjustedGross`/handicap differential. 🟡
- `src/ai.js:242`, `src/ui.js:1383-1387`, `src/oncourse.js:746` — AI/Cloud-Function response text (`html`, `result.data.answer`, `mockAns`) is rendered via `innerHTML`/`renderMarkdownLite`/`markdownToHtml` without first HTML-escaping the raw string before applying markdown substitutions. If the model ever echoes back attacker-influenced content (e.g., a malicious course/round name fed into a prompt), it would be rendered as live HTML in the viewer's browser. 🟡

## 3. console.log statements with user data / internal paths

- `src/auth-v2.js:134` — `console.log("[Auth] User detected:", user.uid, "Fetching doc...")` prints the authenticated user's UID to the browser console on every single login. 🟡
- `src/oncourse.js:900` and `src/oncourse.js:908` — `console.log("Diary uploaded successfully:", url)` and `console.log("[Data Handshake] audioUrl saved to Firestore:", docId)` log the Cloud Storage download URL (which embeds the user's UID and round ID in its path) and the Firestore document ID to the browser console. 🟡
- `src/surveyor.js:139/143/178/195` — logs raw GPS latitude/longitude/accuracy and computed green-center coordinates to the console on every pin capture. Only exposes the surveying user's own location data to their own console, but is unconditional (no dev-only gate). 🟢
- `src/firebase-config.js:15` — the Firebase Web `apiKey` is hard-coded in source. This is standard/expected for Firebase web apps (it identifies the project, isn't a secret, and access is governed by security rules/App Check) — flagged only for completeness, not as a real exposure. 🟢
- `functions/index.js:236/245/332/342` — server-side `console.log` calls embed user UIDs and Firestore paths (`users/${uid}/practice_plans/active`, etc.) into Cloud Logging. Not exposed to end-user browsers, but worth knowing these identifiers persist in centralized logs visible to anyone with Cloud Logging / project viewer access. 🟢
- `src/state.js:78` — `console.log([State] ${prop} changed:, value)` would dump full state objects (including round/profile data) to the console, but it is gated behind a `localhost`/`127.0.0.1` hostname check and won't fire in production. 🟢

## 4. Auth state edge cases

- `src/auth-v2.js:149` — `if (true || userData.isApproved === true || userData.isApproved === undefined || userData.isAdmin === true)`. The `true ||` short-circuits the entire approval check: **every authenticated user is treated as approved** regardless of their actual `isApproved`/`isAdmin` Firestore values. This looks like a leftover debug override that defeats the whole pre-approval/access-control flow described by the surrounding code and comments ("Implicitly approve legacy users and explicit admins" / "Force allowed for verification"). 🔴
- `src/auth-v2.js:217-229` — the `catch` block for a failed `getDoc(users/{uid})` read is labeled "forcing bypass": it unconditionally sets `window.currentUserIsAdmin = true`, hides the auth overlay, and shows the main app. Any transient Firestore error on first login (offline, security-rule misconfiguration, slow connection) silently grants the user **full admin UI visibility** client-side. 🔴
- `src/auth-v2.js:130-242` — `onAuthStateChanged` has no special handling for token-expiry / `auth/user-disabled` / `auth/user-token-expired` style events. When Firebase silently signs a user out mid-session (revoked token, account disabled or deleted from the console), the only visible effect is the `else` branch re-showing the login overlay — there is no message explaining *why*, so a user mid-round on the course would simply see the login screen reappear with no explanation. 🟡
- `src/auth-v2.js` (whole flow) vs. many modules — `AppState.currentUser` is only populated inside the `onAuthStateChanged` callback (after up to 3 retries × 500 ms waiting on `getDoc`). Numerous call sites assume it is already non-null without a guard: `src/ai.js:81`, `src/oncourse.js:1452`, `src/social.js:75/82`, `src/whs.js:373`. If module bootstrapping ever races ahead of (or runs after a delayed) first auth resolution — e.g. a slow connection that exhausts the retry loop — these throw `TypeError: Cannot read properties of null (reading 'uid')`. 🟡
- `src/auth-v2.js:60` — registration only checks `if (!name)`, so whitespace-only names, emoji-only names, or arbitrarily long strings are accepted as `displayName` and persisted — feeding directly into the unescaped-rendering issues catalogued in Area 2. 🟢
